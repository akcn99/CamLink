# 车辆驶入检测功能开发方案

## 1. 目标与范围

本功能的目标是：针对指定摄像头的视频流，在 WebUI 中手动画出一个多边形检测区域，当车辆或二轮车进入该区域时，系统自动发出告警，并通过 Telegram Bot 推送文字和截图。

第一版只解决“实用可落地”的最小闭环，不追求一步到位做成完整 AI 平台。

第一版 MVP 包含：
- 指定摄像头启用检测
- 在画面上绘制多边形检测区域
- 每秒取 1 帧进行检测
- 识别 `car`、`motorcycle`、`bicycle`
- 判断“进入区域”，不处理“离开区域”告警
- 通过 Telegram Bot 推送告警消息和截图
- 在 WebUI 中查看和修改检测配置
- 将驶入事件写入数据库并按时间段查询
- 支持导出 CSV 报表

第一版暂不包含：
- 自定义训练模型
- 精准区分“电动自行车”和“普通自行车”
- 多级告警联动
- GPU 集群或大规模多路并发优化

## 2. 为什么不建议只做画面对比

单纯“每秒抓一张图，然后和上一张图做像素差异”虽然实现快，但误报会非常多，例如：
- 光线变化
- 树影晃动
- 雨雪、夜间噪点
- 行人经过
- 摄像头自动曝光

因此第一版建议采用“低频取帧 + 目标检测 + 轨迹判断”的方案：
- 每秒只处理 1 帧，控制资源消耗
- 用现成检测模型识别车辆目标
- 用目标跟踪判断同一辆车是否从区域外进入区域内

这样比纯画面对比稳定得多，而且更容易解释“为什么触发告警”。

## 3. 如何判断“进入”而不是“出去”

核心不是看单张图片，而是看同一个目标的轨迹。

建议规则如下：
- 每个目标分配一个 `track_id`
- 对每一帧，取检测框的“底边中心点”作为车辆落点
- 判断该点是否位于多边形区域内
- 当某个 `track_id` 的状态从 `outside -> inside` 时，判定为“进入”
- 当状态从 `inside -> outside` 时，视为离开，不告警

为降低误报，再增加两条约束：
- 必须先在区域外连续出现至少 1 次，再进入区域内，才算“驶入”
- 进入区域后，至少连续 2 帧仍在区域内，才真正触发告警

这样可以避免：
- 目标一出现就在区域内，被误认为驶入
- 边缘抖动造成反复触发

## 4. 推荐架构

建议采用“两服务架构”，而不是把 AI 检测全部塞进当前 Go 主程序。

### 4.1 服务划分

1. `CamLink 主服务（Go）`
- 负责现有 WebUI、流管理、鉴权、配置保存
- 新增检测配置页面和 API
- 负责管理哪个摄像头启用检测
- 接收检测事件并展示

2. `Detector 检测服务（Python）`
- 独立 Docker 容器运行
- 负责取帧、目标检测、目标跟踪、进入事件判断
- 调用 Telegram Bot 发送消息
- 将检测结果回传给 CamLink

### 4.2 为什么检测服务建议用 Python

不是因为 Go 不能做，而是因为现成生态在 Python 更成熟：
- OpenCV 取帧成熟
- 现成 YOLO 检测模型容易接入
- 现成目标跟踪工具多
- 调试效率高

这对当前项目最重要：先把功能做出来，后面再考虑是否重构。

## 5. 推荐技术路线

### 5.1 目标检测

第一版建议使用轻量级 YOLO 模型的 CPU 推理版本。

建议策略：
- 输入分辨率先控制在 `640` 左右
- 每秒只处理 `1 FPS`
- 检测类别先限制在：
  - `car`
  - `motorcycle`
  - `bicycle`

说明：
- “电动自行车”在通用模型里通常不会被稳定单独分类
- 第一版先把它归入“二轮车”范围
- 等功能闭环稳定后，再评估是否做自定义分类

### 5.2 目标跟踪

推荐使用轻量目标跟踪器，例如 ByteTrack 类方案。

作用：
- 为同一辆车维持连续 ID
- 用于判断“从外到内”的状态变化
- 用于告警去重

### 5.3 Telegram 推送

告警触发后发送：
- 摄像头名称
- 告警时间
- 告警类型：车辆驶入检测区域
- 抓拍截图
- 可选附加信息：识别类别、置信度

## 6. 第一版数据流

完整链路如下：

1. 用户在 WebUI 里选择某个摄像头，点击“检测区域配置”
2. 页面显示当前视频预览或静态截图
3. 用户用鼠标点击多个点，形成多边形区域
4. 用户保存规则：
   - 启用检测
   - 目标类别
   - 冷却时间
   - Telegram 配置
5. Go 主服务保存配置到 `config.json`
6. Detector 服务读取配置并开始工作
7. Detector 每秒从该路摄像头取 1 帧
8. Detector 检测出车辆目标并分配 `track_id`
9. 若发现同一目标从区域外进入区域内，则生成事件
10. Detector 调用 Telegram Bot API 发送告警
11. Detector 将事件写入事件库，并回传给 Go 主服务，供 UI 显示日志与统计

## 7. 配置设计建议

建议把检测配置挂到每个 `stream/channel` 下，而不是全局单独散落。

示意结构：

```json
{
  "detection": {
    "enabled": true,
    "mode": "vehicle_entry",
    "sample_fps": 1,
    "cooldown_seconds": 30,
    "classes": ["car", "motorcycle", "bicycle"],
    "polygon": [[120, 80], [580, 90], [620, 410], [110, 420]],
    "trigger_consecutive_frames": 2,
    "telegram_enabled": true,
    "telegram_chat_id": "123456789"
  }
}
```

注意：
- `telegram_bot_token` 不建议直接明文写入 `config.json`
- 事件数据库建议默认使用 SQLite 文件，例如 `save/detection-events.db`
- 更推荐放到 Docker 环境变量中
- 例如：`TELEGRAM_BOT_TOKEN=xxx`

## 9. 开发分期

## Phase 1：方案落地与基础框架

目标：把“可运行骨架”搭起来。

交付内容：
- 新增检测配置结构体
- 在 UI 中看到“启用检测”入口
- 新增 Detector 服务目录和 Dockerfile
- 新增 Telegram 配置入口
- 定义 Go 与 Detector 的 API 协议

本阶段不追求真检测，只要：
- 配置能保存
- Detector 容器能启动
- Go 能和 Detector 通信
- 统计页和 CSV 导出接口骨架可访问

验收标准：
- 启动 CamLink + Detector 两个容器成功
- UI 能保存检测开关和区域数据
- Detector 健康检查接口返回正常

## Phase 2：低频取帧与目标检测

目标：真正识别画面中的车。

交付内容：
- Detector 连接 RTSP 地址
- 每秒取 1 帧
- 跑目标检测模型
- 输出检测框、类别、置信度
- 提供调试接口，返回一张带框图片
- 将事件写入数据库的基础能力可用

本阶段先不做“进入判断”，只验证“能否看懂画面里有什么”。

验收标准：
- 指定摄像头能返回带框调试图
- 车辆/二轮车能被基本识别
- CPU 占用可接受

## Phase 3：区域进入判定

目标：从“看见车”升级到“看见车驶入指定区域”。

交付内容：
- 加入目标跟踪 `track_id`
- 加入多边形区域判断
- 加入 `outside -> inside` 状态机
- 加入告警冷却时间
- 避免同一辆车重复推送

验收标准：
- 车辆进入区域时触发一次告警
- 停留在区域内不重复刷屏
- 离开区域不触发“驶入”告警

## Phase 4：Telegram 推送与 UI 完整化

目标：形成完整闭环。

交付内容：
- Telegram Bot 推送文字 + 截图
- 按时间段统计事件数量
- CSV 报表导出
- WebUI 新增测试推送按钮
- WebUI 显示最近事件
- 错误状态提示，例如 Bot Token 无效、RTSP 无法连接

验收标准：
- 用户点击“测试推送”可收到消息
- 实际车辆驶入时 Telegram 能收到截图和时间
- UI 中能看到最近的检测事件

## 10. 本地开发与部署方式

由于你本地没有 Go/Python 开发环境，建议全程用 Docker/OrbStack。

### 10.1 你需要准备的工具

你已经具备：
- mac mini M4
- macOS 15.3
- OrbStack
- Git 仓库本地工作区

后续只需要确认下面两个命令可用：

```bash
docker version
docker compose version
```

如果这两个命令能正常输出版本号，就可以继续。

### 10.2 本地运行方式建议

后续推荐改成 `docker compose` 管理两个服务：
- `camlink-app`
- `camlink-detector`

原因：
- 一个容器负责主程序
- 一个容器负责 AI 检测
- 日志、环境变量、网络更清楚

### 10.3 后续你需要执行的典型命令

启动：

```bash
docker compose up --build
```

后台启动：

```bash
docker compose up -d --build
```

查看日志：

```bash
docker compose logs -f camlink-app
docker compose logs -f camlink-detector
```

停止：

```bash
docker compose down
```

如果以后我让你“重建检测服务”，通常就是执行：

```bash
docker compose up -d --build camlink-detector
```

## 11. Telegram Bot 配置步骤

这部分不复杂，后续上线前你需要做一次。

### 11.1 创建 Bot

1. 在 Telegram 里搜索 `@BotFather`
2. 发送 `/newbot`
3. 按提示输入：
- 机器人名称
- 机器人用户名
4. 创建成功后，BotFather 会返回一个 `Bot Token`

这个 Token 类似：

```text
123456789:AAxxxxxx...
```

### 11.2 获取 Chat ID

1. 先给你创建的 Bot 发一条消息，例如 `hello`
2. 然后通过接口获取更新：

```bash
curl "https://api.telegram.org/bot<你的TOKEN>/getUpdates"
```

3. 返回 JSON 中会看到你的 `chat.id`
4. 这个数字就是推送目标 ID

### 11.3 安全建议

不要把 Token 直接硬编码进仓库。

推荐方式：
- 把 Token 放到 `.env` 或 Docker 环境变量里
- UI 里只填写 `chat_id`

## 12. 资源消耗预估

以下是保守估计，不是严格上限。

单路摄像头，`1 FPS`，`640` 输入，CPU 检测：
- 本地 M4 调试：通常可行
- 云服务器 1 到 2 路：通常可行
- 云服务器 4 路以上：需要实际压测

影响性能的关键因素：
- 摄像头分辨率
- 每秒处理帧数
- 模型大小
- 同时启用检测的摄像头数量

因此第一版必须严格控制：
- `1 FPS`
- 轻量模型
- 只对指定摄像头启用检测

## 13. 风险与取舍

### 13.1 识别“电动自行车”不一定非常准

通用模型通常更擅长：
- `car`
- `motorcycle`
- `bicycle`

“电动自行车”可能被分到 `bicycle` 或 `motorcycle`。

第一版建议接受这个现实：
- 先把目标定义成“车辆/二轮车进入”
- 后续若确实需要细分，再考虑定制模型

### 13.2 夜间与遮挡会影响效果

以下情况会影响准确率：
- 夜视红外
- 逆光
- 大雨
- 车辆重叠遮挡
- 摄像头角度过高或过低

所以第一版应优先选择：
- 视角稳定
- 区域边界清晰
- 目标较大的摄像头先试点

### 13.3 第一版允许“一路摄像头额外增加一个检测连接”

为了尽快落地，第一版 Detector 服务可以直接连接摄像头 RTSP 地址。

优点：
- 架构简单
- 调试快
- 容易独立定位问题

缺点：
- 摄像头会多一个 RTSP 消费者

如果后续发现摄像头连接能力有限，再优化成“复用 CamLink 内部流”。

## 14. 推荐的开发顺序

建议严格按以下顺序推进，不要跳步骤：

1. 写好检测配置结构和方案文档
2. 搭建 Detector 空服务 + Docker 容器
3. 实现测试接口和健康检查
4. 接通 RTSP 取帧
5. 接通目标检测
6. 做区域绘制 UI
7. 做进入状态机
8. 做 Telegram 推送
9. 做事件日志和调试页面

## 15. 对你最重要的说明

你不需要先学会机器学习再开始。

你只需要按阶段理解这几件事：
- 这一路摄像头是否启用了检测
- 区域有没有画对
- 模型有没有识别到目标
- 这个目标是不是从区域外进入区域内
- Telegram 有没有收到消息

后续真正需要你执行的操作，我会用非常具体的形式告诉你，例如：
- 打开哪个文件
- 修改哪一行
- 执行哪条命令
- 看到什么输出算成功
- 出错后看哪个日志

## 16. 下一步建议

正式开发前，建议你先确认下面 5 个产品规则：
- 检测对象是否接受先按“车辆/二轮车”处理，而不是强制区分电动自行车
- 每一路摄像头是否只允许配置 1 个检测区域，还是未来要支持多个区域
- Telegram 是否是第一优先级通知方式
- 第一版是否接受“检测服务直接连接摄像头 RTSP”
- 第一版是否只做单路试点，不同时启用多路检测

在你确认这些规则后，就可以进入 `Phase 1` 实施。
